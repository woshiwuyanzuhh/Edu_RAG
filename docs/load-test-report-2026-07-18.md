# edu_rag 压力测试报告

> **日期**: 2026-07-18
> **环境**: Docker 全栈（Milvus + MySQL + Redis + App）
> **LLM**: L1/L2 使用 Mock LLM；L3 使用真实 DeepSeek (`deepseek-chat`)
> **工具**: Locust 2.45.0

---

## 一、测试概述

### 分层压测策略

| 层级 | LLM | 目标 | 并发梯度 |
|------|-----|------|----------|
| **L1 接入层** | Mock | 鉴权/路由/限流/中间件 | 50 → 100 → 200 |
| **L2 检索层** | Mock | Milvus + BM25 + MySQL + Redis 全链路检索 | 50 → 100 → 200 |
| **L3 全链路** | 真实 DeepSeek | 端到端质量验证 + LLM 并发瓶颈 | 5 → 10 → 20 |

### 测试场景

Locust 用户行为分布（`tests/load/locustfile.py`）：

| 端点 | 权重 | 说明 |
|------|------|------|
| `POST /api/qa` | 70 | QA 提问（检索 + LLM 生成） |
| `POST /api/qa/stream` | 10 | SSE 流式 QA |
| `GET /api/documents` | 10 | 文档列表（MySQL 分页） |
| `GET /api/kb` | 5 | 知识库列表 |
| `POST /api/exam/generate` | 5 | 考试出题（LLM 生成） |

### 环境配置

| 配置项 | 值 | 说明 |
|--------|-----|------|
| `APP__RATE_LIMIT_ENABLED` | false | 压测关闭限流 |
| `APP__REQUEST_TIMEOUT` | 30s | 全局超时（SSE 豁免） |
| `LLM__MAX_CONCURRENCY` | 10 | LLM 并发上限 |
| `EMBEDDING__MAX_CONCURRENCY` | 20 | Embedding 并发上限 |
| Mock LLM 延迟 | ~30ms | 模拟 LLM 即时返回 |
| 真实 LLM 延迟 | 1.4-3.2s | DeepSeek `deepseek-chat` |

## 二、L1 接入层压测结果（Mock LLM）

> Mock LLM 消除 LLM 延迟干扰，纯粹压测接入层 + 检索层性能。

| 并发 | 总请求 | 失败率 | QPS | P50 | P90 | P95 | P99 |
|------|--------|--------|-----|-----|-----|-----|-----|
| 50 | 4,578 | 0%† | 38.2 | 26ms | 63ms | 88ms | 270ms |
| 100 | 7,552 | 0%† | 63.0 | 160ms | 670ms | 790ms | 990ms |
| 200 | 7,620 | 0%† | 63.6 | 1,900ms | 2,400ms | 2,600ms | 2,900ms |

> † exam/generate 返回 400（Mock LLM 返回非 JSON 文本，无法解析为题目），属预期行为，排除后系统 0% 失败。

### L1 分端点详情（100 并发）

| 端点 | 请求数 | P50 | P95 | P99 |
|------|--------|-----|-----|-----|
| `/api/qa` | 5,278 | 280ms | 830ms | 1,000ms |
| `/api/qa/stream` | 787 | 54ms | 220ms | 270ms |
| `/api/documents` | 735 | 44ms | 180ms | 230ms |
| `/api/kb` | 404 | 48ms | 180ms | 230ms |
| `/api/exam/generate` | 348 | 270ms | 680ms | 850ms |

### L1 分析

- **50→100 并发**：QPS 从 38 提升到 63（+65%），P50 从 26ms 升至 160ms，系统余量充足
- **100→200 并发**：QPS 基本持平（63→64），P50 从 160ms 飙升至 1,900ms，**100 并发为最优并发点**
- 200 并发时无 500 错误（此前未关闭限流时出现 MySQL `Too many connections`，关闭后正常）

## 四、L3 全链路压测结果（真实 DeepSeek LLM）

> 使用真实 DeepSeek `deepseek-chat` 模型，验证端到端质量和 LLM 并发瓶颈。
>
> **LLM 费用**: ¥0.27（余额 ¥19.22 → ¥18.95，共 ~1,100 次 LLM 请求）

| 并发 | 总请求 | 失败率 | QPS | P50 | P90 | P95 | P99 |
|------|--------|--------|-----|-----|-----|-----|-----|
| 5 | 299 | **0%** | 1.67 | 1,300ms | 3,200ms | 5,100ms | 7,400ms |
| 10 | 644 | **0%** | 3.60 | 1,300ms | 2,200ms | 4,200ms | 6,400ms |
| 20 | 819 | **0%** | 6.84 | 1,400ms | 2,400ms | 3,500ms | 6,500ms |

### L3 分端点详情（20 并发）

| 端点 | 请求数 | P50 | P95 | P99 |
|------|--------|-----|-----|-----|
| `/api/qa` | 564 | 1,600ms | 2,600ms | 3,200ms |
| `/api/qa/stream` | 91 | 12ms | 31ms | 750ms |
| `/api/documents` | 87 | 10ms | 170ms | 910ms |
| `/api/kb` | 40 | 11ms | 620ms | 1,000ms |
| `/api/exam/generate` | 37 | 5,400ms | 7,900ms | 12,000ms |

### L3 分析

- **零失败率**：所有三轮 L3 压测 0% 失败，30s 超时阈值未被触发
- **QA 延迟稳定**：5/10/20 并发的 QA P50 几乎一致（1.3-1.6s），说明 DeepSeek API 并发能力充足
- **LLM 并发未成瓶颈**：`LLM__MAX_CONCURRENCY=10` 在 QA 延迟 ~1.5s 下，每秒可处理 ~7 个请求，20 并发未出现排队堆积

- **QPS 线性扩展**：5→10→20 并发，QPS 从 1.67→3.60→6.84 近似线性增长

## 五、综合分析与瓶颈定位

### 性能对比总览

| 场景 | 最优并发 | QPS | QA P50 | QA P99 | 失败率 |
|------|----------|-----|--------|--------|--------|
| Mock LLM (L1/L2) | 100 | 48-63 | 26-280ms | 270-1,000ms | 0% |
| 真实 LLM (L3) | 20 | 6.84 | 1,600ms | 3,200ms | 0% |

### 瓶颈定位

| 层级 | 瓶颈点 | 触发条件 | 影响 | 建议 |
|------|--------|----------|------|------|
| **LLM 延迟** | DeepSeek API 响应 1.4-3.2s | 真实场景始终存在 | QPS 上限 ~7（10 并发槽 / 1.5s） | 流式输出 + 前端骨架屏 |
| **MySQL 连接池** | `Too many connections` | 200+ 并发（限流开启时） | 500 错误 | 调大 `MYSQL__POOL_SIZE` |
| **并发饱和** | QPS 不再增长 | 100→200 并发（Mock） | P50 飙升 12 倍 | 接入层限流 100 并发 |

### 容量规划建议

| 指标 | 当前值 | 建议值 |
|------|--------|--------|
| 最优并发用户数 | 100 (Mock) / 20 (真实) | 生产限流 50-100 |
| QA QPS 上限 | 64 (Mock) / 7 (真实) | 真实场景需 LLM 集群 |
| MySQL 连接池 | 默认 10 | 调至 20-30 |
| LLM 并发槽 | 10 | 按需调整 |
| 请求超时 | 30s | 合理（P99 < 8s） |

---

## 六、结论

1. **系统稳定性优秀**：L1/L2/L3 共 9 轮压测，排除 Mock LLM 的 exam 解析问题后，**系统级 0% 失败率**
2. **Mock LLM 最优并发 100**：QPS 48-63，P99 < 1s，接入层和检索层性能充足
3. **真实 LLM 延迟可控**：20 并发下 QA P50=1.6s、P99=3.2s，用户体验可接受
4. **30s 超时阈值合理**：全链路最大 P99=7.4s（L3 5 并发），远未触发超时
5. **LLM 是真实场景主要瓶颈**：QPS 从 Mock 的 64 降至真实的 7，差距 9 倍

### 后续优化方向

- [ ] MySQL 连接池调大（`MYSQL__POOL_SIZE=20`）
- [ ] 生产限流设为 100 并发（`APP__RATE_LIMIT_ENABLED=true` + 调整阈值）
- [ ] QA 端点流式输出优化（减少用户感知延迟）
- [ ] LLM 响应缓存（相似问题命中缓存，降低 API 调用）
