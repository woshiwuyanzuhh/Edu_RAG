## 一、架构设计

### 1.1 四层物理分层 + 共享层 + 可观测层

```
┌──────────────────────────────────────────────────────────────────┐
│                    Orchestration 编排层                           │
│  FastAPI + 中间件链(5) + 4 个 API Router + SPA fallback           │
│  通过 IGenerationService / IIngestionService 门面调用下层          │
├──────────────────────────────────────────────────────────────────┤
│                    Generation 生成层                              │
│  qa_engine / exam_engine / context_pipeline / guardrails          │
│  通过 IRetrievalService 调用检索 — 绝不 import retrieval 内部模块  │
├──────────────────────────────────────────────────────────────────┤
│                    Retrieval 检索层                               │
│  RetrievalService(门面) → recall → rerank → filters → context     │
│  底层: Embedder / VectorStore / BM25 / QueryExpander / RRF fusion │
├──────────────────────────────────────────────────────────────────┤
│                    Ingestion 写入层                               │
│  Pipeline: parse → clean → chunk → filter → embed → vector_store  │
│  策略注册表: 4 Parser + 3 Cleaner + 1 Chunker                     │
├──────────────────────────────────────────────────────────────────┤
│                    Shared 共享层                                  │
│  Config(Pydantic) + MySQL(aiomysql) + Redis + Cache(L1+L2) +      │
│  Exceptions(8种) + Security + ORM/Schemas + logging_config        │
├──────────────────────────────────────────────────────────────────┤
│                    Observability 可观测层                         │
│  Tracer(span) + ContextVar + RAGAS + RetrievalLogger + Feedback   │
└──────────────────────────────────────────────────────────────────┘
```

**这不仅仅是"分层"，而是严格遵循了六边形架构 (Hexagonal Architecture)**：

- 每层之间通过接口 (ABC) 通信，不允许直接 import 实现类
- `IRetrievalService` 是核心"端口"——Generation 层只能通过它调用检索能力
- 任何实现替换（Embedding 换 OpenAI、VectorStore 换 Milvus）都不影响上层代码

### 1.2 12 个接口契约 — 架构的基石

| # | 接口 | 所在层 | 核心方法 | 设计意图 |
|---|------|--------|---------|---------|
| 1 | `IParser` | Ingestion | `parse(file_path) → str` | 解耦文件格式 |
| 2 | `ICleaner` | Ingestion | `clean(text) → str` + `filter_chunks()` | 解耦清洗策略 |
| 3 | `IChunker` | Ingestion | `split(text) → list[str]` | 解耦分块算法 |
| 4 | `IEmbedder` | Retrieval | `embed(list[str]) → list[list[float]]` | **强制批量**，避免 N+1 API 调用 |
| 5 | `IVectorStore` | Retrieval | `search / insert / delete / connect / disconnect / count` | 统一 ChromaDB 和 Milvus 的操作语义 |
| 6 | `IReranker` | Retrieval | `rerank(query, candidates, top_k)` | 解耦重排策略 |
| 7 | `IQueryExpander` | Retrieval | `expand(question, n) → list[str]` | 解耦查询扩展 |
| 8 | `ILLMClient` | Generation | `chat()` + `chat_stream()` | 非流式+流式双通道；流式返回 `AsyncGenerator[str, None]` |
| 9 | `IRetrievalService` | **跨层边界** | `retrieve / retrieve_with_context / build_context_for_exam / build_bm25` | **唯一跨层通道** — Generation 绝不 import retrieval 内部模块 |
| 10 | `IIngestionService` | Ingestion | `ingest / delete_document` | 写入层门面 |
| 11 | `IGenerationService` | Generation | `qa / qa_stream / generate_exam / grade_exam` | 生成层门面 |
| 12 | `IContextProcessor` | Generation | `process(chunks, query) → list[dict]` | 可插拔上下文增强管线步骤 |

### 1.4 六种设计模式的落地

#### 1.4.1 依赖注入 (Dependency Injection) — 核心模式

```python
# RetrievalService 通过构造函数接收所有依赖 — 全部是接口，不是具体类
class RetrievalService(IRetrievalService):
    def __init__(
        self,
        embedder: IEmbedder,           # ← 接口
        vector_store: IVectorStore,    # ← 接口
        llm_client: ILLMClient | None, # ← 可选接口
    ):
        ...

# Generation 层函数接收 IRetrievalService — 不关心内部实现
async def qa_non_stream(
    question: str,
    llm_client: ILLMClient,
    retrieval_svc: IRetrievalService,  # ← 注入接口
    ...
) -> dict:
```

**收益**: 单元测试无需真实数据库、Mock 接口即可；Embedding 从 Ollama 切到 OpenAI 只需改工厂函数；VectorStore 从 ChromaDB 切到 Milvus 只需改一行配置。

#### 1.4.2 策略模式 (Strategy Pattern) + 注册表

```python
# 解析器注册表 — 新增格式只需加一行
PARSER_REGISTRY: dict[str, IParser] = {
    ".pdf": PDFParser(),
    ".docx": DocxParser(),
    ".doc": DocxParser(),
    ".md": MarkdownParser(),
    ".txt": TxtParser(),
}

# 清洗器注册表 — 新增文档类型只需加一行
CLEANER_REGISTRY: dict[str, ICleaner] = {
    "general": BaseCleaner(),
    "education": EducationCleaner(),
    "gaming": GamingCleaner(),
}
```

#### 1.4.3 门面模式 (Facade Pattern)

`RetrievalService` 封装了完整的检索管线：
```
recall → (可选 BM25+RRF) → rerank → filter → build_context
```
对外只暴露 3 个方法：`retrieve()` / `retrieve_with_context()` / `build_context_for_exam()`

#### 1.4.4 模板方法 (Template Method) — 在写入管线中

```python
async def run_ingestion(file_path, doc_id, kb_id, doc_type) -> IngestionResult:
    parser  = PARSER_REGISTRY.get(ext)           # 1. 选解析器
    raw_text = await asyncio.to_thread(parser.parse, file_path)
    cleaner = CLEANER_REGISTRY.get(doc_type)      # 2. 选清洗器
    cleaned_text = await asyncio.to_thread(cleaner.clean, raw_text)
    chunks = await asyncio.to_thread(chunker.split, cleaned_text)  # 3. 切分
    filtered = cleaner.filter_chunks(chunks)       # 4. 过滤
    # 5. 批量 Embedding → 写入向量库 → 同步 BM25
```

管线骨架固定，每一步的具体实现通过注册表切换。所有同步操作通过 `asyncio.to_thread()` 避免阻塞事件循环。

#### 1.4.5 工厂 + 单例 (Factory + Singleton)

```python
# 配置单例
@lru_cache()
def get_settings() -> Settings:
    return Settings()

# Embedder 工厂 — 配置驱动切换
def get_embedder() -> "IEmbedder":
    if settings.embedding.provider == "local":
        return LocalSTEmbedder()
    return OllamaEmbedder()

# VectorStore 工厂
def get_vector_store() -> "IVectorStore":
    if settings.vector_store.provider == "milvus":
        return MilvusStore()
    return ChromaStore()
```

**收益**: 跨请求复用连接池；配置驱动切换实现，业务代码零改动。

#### 1.4.6 ContextVar 上下文传播

```python
_request_id: ContextVar[str] = ContextVar("request_id", default="")

def set_request_id(request_id: str | None = None) -> str:
    rid = request_id or uuid.uuid4().hex[:12]
    _request_id.set(rid)
    return rid
```

跨越 async 边界自动传播 request_id，无需显式传参。Tracer 和日志系统自动获取当前请求 ID 关联所有记录。

#### 1.4.7 依赖注入总汇

| 模式 | 应用位置 | 说明 |
|------|---------|------|
| 构造函数注入 | `RetrievalService.__init__()` | Embedder + VectorStore + LLMClient |
| 函数参数注入 | `qa_non_stream()`, `generate_exam()` | ILLMClient + IRetrievalService |
| 注册表+策略 | `PARSER_REGISTRY`, `CLEANER_REGISTRY` | 扩展名/文档类型 → 实现 |
| 工厂函数+单例 | `get_embedder()`, `get_vector_store()` | 配置驱动，延迟创建 |
| FastAPI Depends | `get_db() → AsyncSession` | 请求级数据库会话 |
| ContextVar | `_request_id` | 跨层传播，无需显式传参 |

## 二、完整链路设计

### 2.1 QA 问答链路（最核心）

以用户提问 **"什么是反向传播？"** 为例，逐层拆解：

```
┌─ HTTP POST /api/qa {"question": "什么是反向传播？", "knowledge_base_id": 24}
│
├─ 中间件链（顺序至关重要）
│   ├─ RequestIDMiddleware    → 注入 X-Request-ID 到 ContextVar（全链路可追踪）
│   ├─ AuthMiddleware         → API Key 校验（开发环境跳过）
│   ├─ RateLimitMiddleware    → 滑动窗口限流（P2-8）
│   ├─ BodySizeMiddleware     → 限制请求体 ≤ 10MB（P2-8）
│   └─ CORS                   → 跨域安全边界
│
├─ Orchestration: api/qa.py → ask_question()
│   ├─ 校验知识库存在性
│   ├─ 多轮对话上下文: session_manager.get_history(session_id)
│   │   └─ Redis (热, TTL 30min) → MySQL (冷, 持久) 双读
│   ├─ gen_svc.qa(question, kb_id, top_k, history)
│   │
│   └─→ GenerationService.qa()
│       │
│       ├─ [Step 0] Guardrails: check_input(question)
│       │   ├─ InputGuard: prompt injection 检测
│       │   └─ block → 直接返回拒绝理由，不执行后续步骤
│       │
│       ├─ [Step 1] retrieval_svc.retrieve_with_context(query, kb_id, top_k=5)
│       │   │
│       │   └─→ RetrievalService.retrieve()
│       │       │
│       │       ├─ Tracer(query="什么是反向传播？")    ← 全链路计时开始
│       │       │
│       │       ├─ span("recall")                     ← 粗排召回
│       │       │   ├─ LLMQueryExpander.expand(query, n=4)
│       │       │   │   ├─ SHA256(query) → 检查 Redis 缓存 (TTL 600s)
│       │       │   │   ├─ 缓存命中 → 直接返回（<5ms）
│       │       │   │   └─ 缓存 miss → LLM 生成 4 个变体:
│       │       │   │       ["什么是反向传播？",
│       │       │   │        "反向传播算法的基本原理与推导",
│       │       │   │        "反向传播在深度神经网络训练中的应用",
│       │       │   │        "反向传播的数学推导与链式法则"]
│       │       │   │
│       │       │   ├─ Embedder.embed(queries)        ← 1 次 API 调用，4 个 1024d 向量
│       │       │   │   ├─ SHA256(str(queries)) → 检查 L1 进程内存缓存 (TTL 60s)
│       │       │   │   └─ 未命中 → Ollama bge-m3 API → 回写 L1 + L2 缓存
│       │       │   │
│       │       │   ├─ VectorStore.search(emb) × 4    ← asyncio.gather 并发
│       │       │   │   └─ 每个 query 取 top_k*2=10 → 共 40 个候选
│       │       │   │
│       │       │   └─ 合并去重（前 120 字符判重）→ 按 score 降序
│       │       │
│       │       ├─ [可选] span("keyword")              ← BM25 混合检索
│       │       │   ├─ bm25.search(query) → 关键词召回
│       │       │   └─ RRF (k=60) 融合向量 + 关键词两路结果
│       │       │       for rank, h in enumerate(vec_hits):
│       │       │           rrf_scores[h.id] += 1.0 / (60 + rank + 1)
│       │       │       for rank, h in enumerate(kw_hits):
│       │       │           rrf_scores[h.id] += 1.0 / (60 + rank + 1)
│       │       │
│       │       ├─ span("rerank")                     ← 精排重排（候选 > top_k 时触发）
│       │       │   ├─ 候选截断: _truncate_head_tail(c.text, 300)
│       │       │   │   └─ 取首 150 字符 + 尾 150 字符（不丢失尾部关键信息）
│       │       │   ├─ LLM 对 Top-10 候选打分（1-10）
│       │       │   ├─ 归一化 + 分数融合:
│       │       │   │   final = 0.7 × LLM_score + 0.3 × vector_score
│       │       │   ├─ LLM 调用失败 → 自动降级: return candidates[:top_k]
│       │       │   └─ JSON 解析失败 → 自动降级: return candidates[:top_k]
│       │       │
│       │       └─ build_context(hits)
│       │           ├─ filter_by_score(min=0.3)      → 过滤低质量片段
│       │           ├─ deduplicate_by_text            → 文本去重
│       │           ├─ limit(max_chunks=10)           → 截断
│       │           └─ lost_in_middle_reorder()       → [best, middle..., second_best]
│       │
│       ├─ [Step 2] _build_messages(context + history + question)
│       │   └─ [
│       │       SystemMessage("你是基于知识库的智能助教..."),
│       │       ...history (多轮对话上下文),
│       │       UserMessage("## 知识库内容\n{ctx}\n\n## 用户问题\n{q}")
│       │     ]
│       │
│       ├─ [Step 3] llm_client.chat(messages, temperature=0.3)
│       │   ├─ with_retry(max_retries=3, backoff=exponential)
│       │   │   ├─ 网络错误 / 5xx          → 指数退避重试 (1s → 2s → 4s)
│       │   │   └─ 401 / 403 / quota超限   → 立即失败，不浪费重试
│       │   └─ → "反向传播（Backpropagation）是训练人工神经网络的
│       │           核心算法，由 Rumelhart、Hinton 和 Williams 于 1986 年
│       │           提出。其核心思想是..."
│       │
│       ├─ [Step 4] Guardrails: check_output(answer)
│       │   ├─ OutputGuard: 引用验证 + 幻觉检测
│       │   └─ block → 替换为拒绝理由
│       │
│       └─ return {"question": ..., "answer": ..., "sources": [...]}
│
├─ 保存对话历史: session_manager.append_message("user", question)
│              + session_manager.append_message("assistant", answer)
├─ 记录检索日志: retrieval_logger.log(query, chunks, answer) → 离线评估数据
│
└─ APIResponse {
    "success": true,
    "data": {
      "question": "什么是反向传播？",
      "answer": "反向传播是训练神经网络的核心算法...",
      "sources": [
        {"doc_id": 15, "chunk_index": 3, "score": 0.92, "text_preview": "..."},
        ...
      ],
      "session_id": "abc123"
    }
  }
```

### 2.2 流式 QA 链路的差异

```
POST /api/qa/stream {"question": "...", "knowledge_base_id": 24}
  │
  ├─ 检索阶段（同上，完全复用 retrieve_with_context）
  │
  └─ llm_client.chat_stream(messages)
      └─ Thread(producer) + asyncio.Queue(consumer)
          │  ← 同步 API stream → loop.call_soon_threadsafe → Queue
          │  ← 异步生成器从 Queue 消费
          │
          ├─ yield "data: 反\n\n"
          ├─ yield "data: 向\n\n"
          ├─ yield "data: 传播\n\n"
          ├─ ...
          ├─ yield "data: {\"type\":\"sources\",\"data\":[...]}\n\n"  ← 末尾返回引用
          └─ yield "data: [DONE]\n\n"
```

**关键设计**: 流式末尾通过 `[SOURCES]` JSON 事件返回引用来源，前端在流式渲染结束后展示引用卡片。实现**体验优先 + 可信度保障**。

### 2.3 文档写入链路

```
POST /api/documents/upload (multipart form: file + kb_id + doc_type)
  │
  ├─ 保存文件到 data/{kb_id}/{doc_id}_{filename}
  ├─ MySQL 创建 document 记录 (status=pending) → 返回 doc_id
  │
  ├─ ingestion_service.ingest(file_path, doc_id, kb_id, doc_type)
  │   │
  │   └─→ run_ingestion()
  │       ├─ 1. parse:  PARSER_REGISTRY[ext].parse(file_path)
  │       │   ├─ PDFParser  → PyMuPDF (fitz) 提取文本
  │       │   ├─ DocxParser → python-docx (含表格提取)
  │       │   ├─ MarkdownParser → markdown 库 (代码块保留)
  │       │   └─ TxtParser → 内置 open() + UTF-8/GBK 自动检测
  │       │   └─ 全部通过 asyncio.to_thread() 执行（不阻塞事件循环）
  │       │
  │       ├─ 2. clean: CLEANER_REGISTRY[doc_type].clean(raw_text)
  │       │   ├─ BaseCleaner (8 步管线):
  │       │   │   ① 去控制字符（保留 \n\t）  ② 去零宽字符和 BOM
  │       │   │   ③ 全角 ASCII → 半角      ④ 移除 URL 和邮箱
  │       │   │   ⑤ 移除 HTML 标签和实体     ⑥ 清理 Markdown 链接
  │       │   │   ⑦ 规范化换行（最多 2 连续） ⑧ 规范化空白
  │       │   ├─ EducationCleaner: BaseCleaner + 去除页码/水印/参考文献行
  │       │   └─ GamingCleaner:    BaseCleaner + 去除论坛签名/广告/表情刷屏
  │       │
  │       ├─ 3. chunk: chunker.split(cleaned_text)
  │       │   └─ RecursiveChunker 分隔符优先级:
  │       │       \n\n → \n → 。→ . → ！→ ？→ ；→ 空格 → 字符级硬切
  │       │       └─ 短块自动合并 + chunk_size=800 + overlap=100
  │       │
  │       ├─ 4. filter: cleaner.filter_chunks(chunks)
  │       │   ├─ min_len ≥ 15 字符
  │       │   ├─ CJK + Latin 字符比例 ≥ 0.2（过滤纯符号/乱码）
  │       │   └─ Jaccard 相似度去重（n=2 gram, 阈值 0.8）
  │       │
  │       ├─ 5. embed: Embedder.embed(chunk_texts)  ← 批量 API 调用
  │       │
  │       ├─ 6. vector_insert: VectorStore.insert(VectorItem[])  ← 写入向量库
  │       │
  │       └─ 7. bm25_build: RetrievalService.build_bm25(documents)
  │           └─ 保持 BM25 索引与向量库同步
  │
  ├─ 更新 MySQL → document status=completed
  ├─ 失效缓存: cache.invalidate(f"edu_rag:*:{kb_id}:*")
  │   └─ 清除 Redis 中该知识库的所有缓存 key
  │
  └─ APIResponse {"success": true, "data": {"chunk_count": 24, "total_chars": 15680}}
```

### 2.4 出题 + 批改链路

#### 出题

```
POST /api/exam/generate {"knowledge_base_id": 24, "question_type": "mixed",
                          "question_count": 5, "difficulty": "medium"}
  │
  ├─ retrieval_svc.build_context_for_exam(kb_id)
  │   └─ 多角度并行召回（3 路 asyncio.gather）
  │       ├─ recall("关键概念 定义 原理")  → top_k=5   (概念角度)
  │       ├─ recall("方法 步骤 流程")      → top_k=5   (方法角度)
  │       └─ recall("应用 案例 示例")      → top_k=5   (应用角度)
  │       → 合并去重 → 按 score 降序 → 取 top-10 → 构建上下文
  │
  ├─ 构造 LLM prompt: EXAM_GENERATE_PROMPT.format(context, type, count, difficulty)
  │
  ├─ llm_client.chat(temperature=0.8, max_tokens=4096)
  │   └─ temperature=0.8 保证题目多样性（比 QA 的 0.3 高得多）
  │
  ├─ 鲁棒 JSON 解析:
```

## 三、项目亮点

### 🏆 亮点 1: 严格的分层解耦 — 工业级架构

这不是"把代码分几个文件夹"的表面分层。真正的约束是：

```python
# ✅ Generation 层只能这样调用检索:
from src.interfaces.retrieval_service import IRetrievalService
result = await retrieval_svc.retrieve_with_context(query, kb_id)

# ❌ 绝不能这样:
from src.retrieval.recall import recall        # 禁止!
from src.retrieval.rerank import llm_rerank    # 禁止!
from src.retrieval.filters import build_context # 禁止!
```

**这意味着**: 如果明天要把整个 retrieval 层换成 Elasticsearch + Cohere Rerank，Generation 层一行代码都不需要改。这就是 SOLID 原则中 "D" (依赖倒置) 的完美落地。

### 🏆 亮点 2: 两阶段检索 + 分数融合 + 优雅降级

```
Recall (向量粗排)                    Rerank (LLM精排)
  速度: 毫秒级                        速度: 秒级
  精度: 中等（语义近似）              精度: 高（深度理解）
  候选: 40 条                         候选: 5 条
        ↓                                  ↓
        └──── 取 Top-10 送入 LLM 打分 ─────┘
                  ↓
        final = 0.7 × LLM_score + 0.3 × vector_score
                  ↓
        LLM 调用失败 → 自动降级: return candidates[:top_k]
        JSON 解析失败 → 自动降级: return candidates[:top_k]
```

**精妙之处**:

- 不是简单的 LLM 重排后直接取 Top-K，而是 **LLM 分数和向量分数融合**，保留了两路独立的信号
- LLM 解析失败时 **自动降级** 返回原始排序，用户体验不受影响
- Rerank 只在 `candidates > top_k` 时才触发，避免对少量候选做无意义的重排
- **两阶段是 RAG 领域精度/延迟比最优方案**——向量检索快但精度有限，LLM 精排准但全量太慢

### 🏆 亮点 3: Lost in the Middle 上下文重排

```
原始排序: [best, 2nd, 3rd, 4th, 5th, 6th, 7th]
            ↓ Lost in the Middle 重排
重排后:   [best, 3rd, 4th, 5th, 6th, 7th, 2nd]
           ↑___首部(注意力最高)___↑          ↑___尾部(注意力次高)___↑
```

基于 **Liu et al. (2023)** 的研究: LLM 对长文本的注意力呈现 U 型曲线——开头和结尾的信息获得高注意力，中间的信息被"稀释"。本项目把最相关的内容放首位、次相关的放末位，较少相关的放中间。当 `len(hits) ≤ 3` 时跳过重排，避免无意义操作。

### 🏆 亮点 4: 查询扩展 + 批量 Embedding — API 级别的性能优化

```python
# IEmbedder 接口强制批量调用
class IEmbedder(ABC):
    @abstractmethod
    async def embed(self, texts: list[str]) -> list[list[float]]:
        """接收文本列表，返回嵌入向量列表。必须批量调用。"""
        ...

# 调用侧: 查询扩展 → 批量 Embedding → 并发检索
queries = await expander.expand("什么是反向传播？", n=4)
# → ["原始", "基本原理", "数学推导", "应用场景"]

embeddings = await embedder.embed(queries)    # ← 1 次 API 调用！
results = await asyncio.gather(*[             # ← 4 路并发检索
    vector_store.search(emb) for emb in embeddings
])
```

**IEmbedder 接口强制 `embed(list[str])` 而非 `embed(str)`**，从接口层面杜绝了 N+1 调用问题。这是一个"通过 API 设计防止性能问题"的绝佳范例。

### 🏆 亮点 5: 三级降级策略 — 韧性的系统设计

```
安全优先 > 可用性优先 > 体验优化

┌─ Rerank LLM 调用失败       → 降级: return candidates[:top_k]      (体验降级)
├─ Rerank JSON 解析失败       → 降级: return candidates[:top_k]      (体验降级)
├─ Query Expand LLM 失败     → 降级: return [original_question]     (体验降级)
├─ Redis 不可用              → 降级: 跳过缓存，直接调 API             (可用性降级)
├─ VectorStore 连接失败      → 降级: 启动不阻止，运行时报错           (可用性降级)
├─ Embedding 缓存 miss       → 降级: 直接调 API                      (可用性降级)
├─ Guard 非安全类异常         → 降级: 放行                            (可用性优先)
└─ Guard 安全类异常           → 拒绝: block                          (安全优先)
```

**核心逻辑**: 安全类 Guard (`is_blocking=True`) 异常时拒绝——安全不可妥协；非安全类 Guard 异常时放行——可用性优先。Rerank 失败不影响回答质量（只是精度稍降），但安全 Guard 失败必须阻止。

### 🏆 亮点 6: 流式 SSE + 线程桥接 + 引用后置

```python
# 同步 API stream → asyncio Queue → AsyncGenerator → SSE Response
def _produce():                      # 在 Thread 中运行
    for chunk in sync_api.stream():
        loop.call_soon_threadsafe(    # ← 跨线程安全写入
            queue.put_nowait, chunk
        )

async def chat_stream():             # AsyncGenerator
    while True:
        chunk = await queue.get()    # ← 异步消费
        yield chunk
```

流式末尾返回引用来源的设计:
```
data: 机器学习是人工智能的重要分支...

data: {"type":"sources","data":[
  {"doc_id":15,"chunk_index":3,"score":0.92,"text_preview":"..."},
  {"doc_id":12,"chunk_index":1,"score":0.87,"text_preview":"..."}
]}

data: [DONE]
```

这解决了 RAG 产品的核心矛盾：**流式 = 体验好但缺乏可信度**。通过 `[SOURCES]` 后置事件，前端可以在流式渲染结束后展示引用卡片。

### 🏆 亮点 7: 可插拔上下文增强管线 (OCP 落地)

```python
# ContextPipeline = 策略模式的链式组合
pipeline = ContextPipeline([
    RelevanceFilter(llm_client),      # 步骤1: LLM 过滤低相关性 chunk
    SemanticCompressor(llm_client),   # 步骤2: 压缩长 chunk 为摘要
    LostInMiddleReorder(),            # 步骤3: 重排
])

```

**开闭原则 (OCP) 的完美落地**: 新增一个上下文处理步骤 = 写一个新类实现 `IContextProcessor` + 注册到管线，现有代码零改动。每个步骤失败时自动捕捉异常并跳过，不影响后续步骤执行。

### 🏆 亮点 8: Guardrails 安全链 — 安全与可用性的精确平衡

```python
chain = GuardrailChain([
    InputGuard(),     # Prompt injection 检测 → block
    RefuseGuard(),    # 低置信度拒答 → block  
    OutputGuard(),    # 引用验证 + 幻觉检测 → block
])

# 内部降级逻辑:
if guard.is_blocking:   # 安全类 Guard 异常
```

任何一个 Guard 返回 `block` → 立刻拒绝，不执行后续 Guard。链式短路 + 差异化降级策略。

### 🏆 亮点 9: 全链路追踪 + 评估数据闭环

```
┌─ Tracer (实时)
│   ├─ span("recall"): 耗时 + 候选数 + 查询扩展数
│   ├─ span("rerank"): 耗时 + 精排后数量 + LLM/向量分数
```

### 🏆 亮点 10: RRF 混合检索 — 零参数融合

```python
# Reciprocal Rank Fusion (k=60) — Cormack et al., 2009
for rank, hit in enumerate(vec_hits):
    rrf_scores[hit.id] += 1.0 / (60 + rank + 1)
for rank, hit in enumerate(kw_hits):
    rrf_scores[hit.id] += 1.0 / (60 + rank + 1)

# 按 RRF 分数降序排列
fused = sorted(rrf_scores.items(), key=lambda x: x[1], reverse=True)
```

向量检索（语义匹配）+ BM25（关键词匹配）通过 **RRF 无参数算法** 融合：

- 不需要调权重
- 不需要训练数据
- 不需要标注
- 自动处理两路结果有重叠和无重叠的情况
- 比简单的线性加权更稳健，是信息检索领域的经典算法

## 总结

**edu_rag v2.0** 是一套严格遵循 SOLID 原则、面向接口编程、充分解耦的四层 RAG 架构。从 12 个接口契约到 6 种设计模式，从两阶段检索到 Lost-in-Middle 重排，从三级降级策略到 Guardrails 安全链，从全链路追踪到评估数据闭环——几乎每一个 RAG 领域的工业最佳实践都被正确落地了。

**核心价值**:

- 任何组件（Embedding / LLM / VectorStore / Parser / Cleaner）都可以 **独立替换** 而不影响其他层
- 两阶段检索 + Lost-in-Middle + RRF 混合检索构成 **检索核心竞争力**
- Tracer + RAGAS + Feedback 提供 **持续优化的量化基础**
- 流式 SSE 双通道兼顾 **用户体验和可信度**
- 多级缓存 + 分级降级确保 **系统韧性**

**架构评分: 4.8/5** — 剩余的 0.2 分留给内容压缩、文档级 ACL 和 Prometheus 仪表盘。
