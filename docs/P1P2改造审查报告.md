### P1 — 严重（1 项，已修复）

| # | 问题 | 位置 | 状态 |
|---|---|---|---|
| P1-1 | ARQ worker 独立进程未初始化 MySQL/Redis/向量库/BM25 连接，导致任务执行时 `_session_factory is None` 抛 RuntimeError，worker 完全无法工作 | `src/orchestration/worker/config.py`（原 WorkerSettings 缺 on_startup） | ✅ 已修复：新增 `on_startup`/`on_shutdown` 钩子，初始化全部依赖并对齐 app lifespan |

**修复说明**：worker 是独立进程，不执行 app 的 lifespan。原 WorkerSettings 只有 functions + redis_settings，启动后 `_session_factory`（mysql.py 模块级）为 None，tasks.py 第 30 行 `if _session_factory is None: raise RuntimeError` 必然触发。已加 `on_startup` 调用 `init_mysql` + `redis_client.connect` + `get_vector_store().connect` + `load_all_bm25_from_db`，`on_shutdown` 释放连接。

### P2 — 重要（8 项）

| # | 问题 | 位置 | 影响 | 建议 |
|---|---|---|---|---|
| P2-1 | BM25 persist 多 worker 竞争 | `keyword/__init__.py:persist_bm25_to_db` | `async_ingestion=false`（开发模式）+ 多 uvicorn worker 时，多个 app worker 并发写同一 kb 的 BM25 索引到 MySQL，互相覆盖 | 生产用 `async_ingestion=true`（C5 队列单写消除竞争）；开发模式避免多 worker；或 persist 加乐观锁（version 字段） |
| P2-2 | /health 探活开销 | `app.py:health_check` | 每次探活调 `vs.count()`，Milvus 会触发 flush（`milvus.py:166`），5s 一次的 LB 探活可能影响写入性能 | 改用轻量探活：Milvus 检查 `_connected` 标志或 `ping`；或探活频率降到 10s |
| P2-3 | 业务指标未埋点 | `middleware/metrics.py` | 定义了 QA_REQUESTS/RETRIEVAL_LATENCY 等指标，但 qa.py/exam_engine/documents.py 未调用 `.inc()/.observe()`，/metrics 无业务数据 | 在关键路径埋点（qa.py ask、exam_engine、documents.py upload） |
| P2-4 | 删除路径双重删文件 | `documents.py:delete_document` | `storage.delete(doc.file_path)` 后 `delete_document_resources` 内部再 `os.remove(file_path)`，LocalStorage 时第二次删不存在的文件抛错（被外层 try/except 吞，但可能影响向量删除顺序） | 分离职责：storage.delete 专删文件，delete_document_resources 专删向量（file_path 参数容错或移除文件删除） |
| P2-5 | failover 无冷却期 | `scripts/failover_check.py` | 触发切换后 `failures=0`，若本地仍不可用，下个周期再次触发切换（重复调 DNS/LB API） | 触发后加冷却期（如 5min 内不重复触发） |
| P2-6 | Dockerfile 用 `uvicorn` 而非 `uvicorn[standard]` | `docker/Dockerfile:12` | pyproject 是 `uvicorn[standard]`（含 uvloop/httptools/websockets），Dockerfile 裸 `uvicorn` 缺失高性能依赖 | Dockerfile 改 `uvicorn[standard]` |
| P2-7 | requirements.txt 未被 Dockerfile 使用 | `docker/Dockerfile:11-18` | P1-4 生成了锁文件（384 行版本锁定），但 Dockerfile 用 `uv pip install` 显式列表安装，未用锁文件，镜像版本不可复现 | Dockerfile 改 `COPY requirements.txt . && uv pip install --system -r requirements.txt`，或 `uv pip install --system .` |
| P2-8 | worker 任务重试时文件已清理 | `worker/tasks.py:55-58` | 任务失败时 `os.remove(doc.file_path)` 清理文件，ARQ 重试时文件不存在，ingest 再次失败 | 重试前检查文件存在；或失败不清理文件（留手动处理），仅最终失败清理 |

## 三、架构合理性评估

### 优点
1. **无状态化推进到位**：会话/考试/BM25 全外部化到 MySQL，L1 缓存可由 L2 Redis 回填，应用层基本无状态，支持水平扩展
2. **异步队列设计合理**：ARQ + Redis broker，文档入库与 API 解耦，单写消除 BM25 竞争，符合工业主流
3. **存储抽象解耦**：storage.py 让本地/对象存储可切换，为多实例/容灾奠基
4. **Milvus 代码层早已就绪**：工厂模式 + 配置开关，切换成本低
5. **容灾方案完整**：检测→切换→回切→演练全流程文档化，RTO/RPO 明确

### 待改进
1. **指标埋点缺失**（P2-3）：监控基础设施就位但无数据，需补埋点
2. **worker 重试鲁棒性**（P2-8）：失败清理与重试冲突
3. **Dockerfile 与锁文件脱节**（P2-7）：镜像不可复现，违背锁文件初衷
