## 一、项目整体评价

### 1.1 项目概览

edu_rag 是一套基于四层 RAG 架构的智能教育题库系统，支持多格式文档解析、两阶段语义检索、智能出题、四维度自动批改、多轮对话问答与 SSE 流式输出。

| 维度 | 技术选型 |
|---|---|
| 后端 | FastAPI 0.115+（async）、SQLAlchemy 2.0 async + aiomysql、Redis、ChromaDB / Milvus |
| AI | Ollama bge-m3 + OpenAI 兼容 LLM（DeepSeek）、自研 Tracer + RAGAS |
| 前端 | Vue 3.5 + Vite 5 + TypeScript 5.5 + Ant Design Vue 4.2 + Pinia |
| 测试 | 226 用例，unit/integration/quality 三级分层 |

### 1.2 优点

- **架构层次清晰**：严格遵循六边形架构，`src/` 划分为 `interfaces/`、`ingress/`、`retrieval/`、`generation/`、`orchestration/`、`shared/`、`observability/` 七个顶层包，13 个 ABC 接口作为跨层契约。
- **解耦到位**：Generation 不直接 import retrieval 内部模块，`IRetrievalService` 作为唯一跨层端口（`retrieval/service.py:27-51`）。
- **设计模式落地**：依赖注入、工厂单例、策略注册表（`PARSER_REGISTRY`/`CLEANER_REGISTRY`）、门面、模板方法均有文档对照。
- **文档体系完备**：README、ARCHITECTURE、CLAUDE、3 份中文架构文档 + Makefile + start/stop 脚本，工程化程度高。
- **代码可读性优秀**：模块级 docstring 普遍高质量，PEP 604/585 现代类型注解全面，`@traced` AOP 装饰器自动埋点。
- **测试分层规范**：unit 下按 src 结构镜像，`conftest.py` 提供 `mock_env` 隔离 .env 与全局状态清理。

### 1.3 问题

| 问题 | 位置 | 严重度 |
|---|---|---|
| LLM 客户端模块双入口 | `src/providers/llm/` 与 `src/generation/llm/`（后者已退化为兼容垫片） | 中 |

| 文档接口数量自相矛盾 | README 称 12 个、ARCHITECTURE 标题 9 个正文列 12 个、CLAUDE 写 14 个，实际 13 个 | 低 |

## 二、安全漏洞检查

### 2.1 严重（Critical）

#### 🔴 C1. 真实 LLM API Key 硬编码
- **位置**：`.env:7` `LLM__API_KEY=sk-8d15888e0bbb4659bcabeb3ffa70e2f6`
- **描述**：明文存储 DeepSeek 真实 API Key。虽然 `.gitignore:26` 已排除且 git 历史无泄露，但本地磁盘明文存储，本机任何进程均可窃取。
- **修复**：立即在 DeepSeek 控制台轮换密钥；改用系统密钥管理（Windows Credential Manager / Vault / 环境变量注入，不落盘）。

#### 🔴 C2. MySQL 弱口令
- **位置**：`.env:29` `MYSQL__PASSWORD=123456`
- **描述**：root 账户弱口令。`src/shared/security.py:12` 仅告警不阻断。
- **修复**：更换强密码；生产用非 root 业务账户并最小化权限。

#### 🔴 C3. API 鉴权默认关闭（全部端点公开）
- **位置**：`src/orchestration/middleware/auth.py:36-38`
- **描述**：当 `APP__API_KEY` 为空时直接放行所有请求。`.env:47` 该项被注释，开箱状态下文档上传、出题、批改、知识库 CRUD、反馈读取等所有接口对任何人开放，可被未授权调用消耗 LLM 配额或删除数据。
- **修复**：生产强制设置 `APP__API_KEY`（≥32 字符）；启动时若 `debug=False` 且 key 为空应直接拒绝启动（fail-closed）。

### 2.2 高危（High）

#### 🟠 H1. 前端存储型 XSS（Markdown 未净化 + v-html）
- **位置**：`frontend/src/views/QAPage.vue:243` 与 `:378`
- **描述**：LLM 回答经 `marked@12`（默认不净化）转 HTML 后直接 `v-html` 渲染。攻击者可通过上传含 `<img onerror=...>` 的文档，诱导问答后在前端执行任意脚本。
- **修复**：引入 `dompurify`，`return DOMPurify.sanitize(marked.parse(text))`。

#### 🟠 H2. docker-compose 默认弱密码 + Redis 无密码
- **位置**：`docker/docker-compose.yml:17,45,62`
- **描述**：MySQL 默认密码 `edu_rag_2024`；Redis 未设 `REDIS_PASSWORD` 时无认证运行；MySQL/Redis/Ollama 端口直接映射宿主机。
- **修复**：移除默认密码占位；Redis 始终启用 `requirepass`；生产不对宿主机暴露数据库端口。

#### 🟠 H3. 依赖未锁版本、无锁文件
- **位置**：`pyproject.toml:6-27`
- **描述**：所有依赖 `>=` 无上界，仓库无 `poetry.lock`/`requirements.txt`。构建不可复现，可能拉入漏洞版本。
- **修复**：生成并提交锁文件（`uv lock` / `pip-compile`）；定期 `pip-audit`/`safety` 扫描。

### 2.3 中危（Medium）

| 编号 | 问题 | 位置 | 修复方向 |
|---|---|---|---|
| M1 | SPA catch-all 路径遍历隐患 | `src/orchestration/app.py:196` | `resolve()` 后断言仍处于 `FRONTEND_DIST` 内 |

### 2.5 通过项（无问题）

- ✅ **SQL 注入**：全部使用 SQLAlchemy 2.0 ORM 参数化查询，未发现原始 SQL 拼接。
- ✅ **命令注入**：全目录无 `subprocess`/`os.system`/`eval`/`exec`/`pickle.load` 调用。
- ✅ **敏感信息脱敏**：`config.py` 用 `SecretStr`，`.gitignore` 正确排除 `.env`，git 历史无泄露。
- ✅ **全局异常处理**：`error_handler.py:67` 捕获 `Exception` 返回通用 500，不泄露堆栈。

## 四、性能瓶颈

| 问题 | 位置 | 影响 | 优化方向 |
|---|---|---|---|
| **上传文件全量读入内存** | `documents.py:51` `content = await file.read()` | 50MB 文件占内存 | `aiofiles` 分块写盘 |
| **同步 `os.remove` 在 async 路径** | `knowledge.py:96-98` 循环逐个同步删文件 | 阻塞事件循环 | `asyncio.to_thread` 或 `gather` |
| **BM25 增删全量重建** | `bm25.py:104,114` 每次 add/remove 重建 `BM25Okapi` | O(n) 开销 | 标记删除 + 延迟批量 rebuild |
| **逐块串行 LLM 压缩** | `compressor.py:22-42` for 循环逐块 await | 管线慢 | `asyncio.gather` 并发（注意限流） |
| **`redis.keys(pattern)` 阻塞** | `cache.py:155` | 生产 Redis 阻塞 | 改用 `SCAN` 迭代 |
| **同步文件写在 async QA 路径** | `retrieval_logger.py:61` 同步 `open+write` | 阻塞事件循环 | `aiofiles` |
| **Milvus 每次 insert 都 flush** | `milvus.py:101` | flush 昂贵 | 批量延迟 flush |
| **session 全量覆盖写** | `session.py:77-81` 每次 append 都 select 全量 + 覆盖 JSON | 浪费 IO | SQL JSON_APPEND 或增量更新 |
| **盲删 10000 ID fallback** | `ingress/service.py:125-127` `range(0,10000,1000)` 硬编码 | 浪费查询 | 先查实际 ID 再删 |
| **全局可变状态无上限** | `keyword/__init__.py:13` `_bm25_indexes`、`cache.py:110` `_l1` | 长期运行内存膨胀 | 加 maxsize + LRU 淘汰 |

## 六、总结

edu_rag 在**架构设计**层面达到了较高水准——四层分层、接口契约、设计模式落地、文档体系完整、测试分层规范，体现了成熟的工程思维。**安全层面**有基本意识（Pydantic 验证、ORM 参数化、SecretStr 脱敏、启动期自检），但存在 3 个严重漏洞（API Key 泄露、弱口令、鉴权默认关闭）必须立即修复，否则生产部署风险极高。**代码质量**整体良好，但有约 200 行死代码、若干重复逻辑和 10 个性能瓶颈待清理。

**核心建议**：先修安全（P0/P1），再清冗余（P2），最后演进架构（P3）。项目具备良好的可演进基础，按本报告优先级推进即可稳步提升至生产可用状态。
