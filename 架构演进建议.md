# edu_rag v2.0 架构演进建议

> 以下建议基于 RAG 系统工业最佳实践（参考 LangChain、LlamaIndex、RAGFlow 等生产级框架的设计范式），针对现有四层架构在功能扩展过程中的**架构方向**给出指导。

------

## 一、核心架构原则（贯穿全部修改）

### 原则 1：层职责必须正交，禁止出现"上帝层"

当前四层架构的最大风险是 **Retrieval 层正在膨胀为"上帝层"**。随着 BM25、HyDE、语义压缩、相关性验证、Parent Document 等功能的加入，Retrieval 层将承担过多职责。

**工业界标准分层模型（5 阶段管线）：**

```
┌─────────────────────────────────────────────────────────────────┐
│                     RAG Pipeline (5 Stages)                     │
├──────────┬──────────┬──────────────────┬───────────┬────────────┤
│ Indexing │ Retrieval│ Post-Retrieval   │ Generation│ Response   │
│ 索引构建 │ 召回+重排 │ 上下文增强/验证  │  生成回答  │  后处理    │
├──────────┼──────────┼──────────────────┼───────────┼────────────┤
│Ingestion │Retrieval │  ⚠️ 当前缺失层   │Generation │Orchestration│
│  Layer   │  Layer   │  需要明确归属     │  Layer    │   Layer    │
└──────────┴──────────┴──────────────────┴───────────┴────────────┘
```

**架构建议：** 不需要新增第五层，但必须在 Retrieval 和 Generation 之间明确一个**逻辑上的"Post-Retrieval / Context Augmentation"阶段**。这个阶段的所有操作（语义压缩、相关性验证、Parent Document 替换、引用映射）在**职责上属于 Generation 层的上下文构建前置步骤**，而不是 Retrieval 层的延伸。

**判断标准：** 如果一个操作需要理解 query 的语义并调用 LLM，它就不属于 Retrieval。Retrieval 只负责"找到候选"，不负责"理解候选"。

------

### 原则 2：单向依赖链 + 接口隔离

```
Ingestion → Retrieval → Generation → Orchestration
    ↑            ↑            ↑
    └────────────┴────────────┴─── 只依赖接口，不依赖实现
```

**架构建议：**

- **每个层必须且只能暴露一个 Facade 接口**（如 `IIngestionService`、`IRetrievalService`、`IGenerationService`）
- **上层只能通过接口调用下层**，禁止 `from src.xxx.internal_module import ...`
- **层内部可以任意组织**（子模块、子包、注册表），但对外只暴露 Facade
- **新增功能时，优先扩展现有接口的方法**，而非新增跨层调用路径

当前缺失的 `IIngestionService` 和 `IGenerationService` 必须补齐，否则随着功能增加，Orchestration 层会退化为"直接拼装各层内部模块"的胶水层，丧失架构意义。

------

### 原则 3：所有增强必须是"可插拔管线"（Pluggable Pipeline）

工业级 RAG 系统的核心设计模式是 **Pipeline + Step**，而非硬编码的 if-else 链。

**架构建议：** 将检索后处理和生成前处理设计为**可配置的步骤链**：

```
Retrieval 返回 chunks
  │
  ▼
┌─────────────────────────────────────────────┐
│           Context Augmentation Pipeline      │
│                                              │
│  Step 1: RelevanceFilter    [on/off]         │
│  Step 2: ParentDocResolver  [on/off]         │
│  Step 3: SemanticCompressor [on/off]         │
│  Step 4: LostInMiddleReorder [always on]     │
│  Step 5: TruncationBySentence [always on]    │
│                                              │
└─────────────────────────────────────────────┘
  │
  ▼
Generation (Prompt 构建 + LLM 调用)
```

每个 Step 实现统一的 `IContextProcessor` 接口：

```python
class IContextProcessor(Protocol):
    async def process(self, chunks: list[Chunk], query: str) -> list[Chunk]: ...
```

**好处：**

- 新增功能 = 新增一个 Step 类 + 注册到管线，不改现有代码
- 步骤顺序可配置，开关可配置
- 每个 Step 可独立测试、独立替换
- 符合开闭原则（对扩展开放，对修改关闭）

------

## 二、各层架构演进建议

### Ingestion 层

**当前问题：** 同步管线，缺乏状态管理和异步能力。

**工业实践方向：**

| 维度     | 当前状态      | 目标状态                                                     |
| -------- | ------------- | ------------------------------------------------------------ |
| 执行模式 | 同步 pipeline | 支持异步任务（大文档后台处理）                               |
| 索引构建 | 只写向量      | 多索引并行写入（向量 + BM25 + 全文）                         |
| 文档结构 | 扁平 chunks   | 保留层级关系（Parent-Child）                                 |
| 状态追踪 | 无            | 文档状态机（Uploading → Parsing → Chunking → Indexing → Ready / Failed） |

**架构建议：**

- Ingestion 的内部 pipeline 应该是一个 **DAG（有向无环图）**，而不是线性链。向量写入和 BM25 写入应该是并行的两个分支，而非串行。
- 新增 `DocumentStatus` 状态枚举，持久化到数据库。Orchestration 层通过查询状态获知处理进度，而非同步等待。
- Parser 和 Cleaner 已经用了注册表模式（✅ 好的），Chunker 也应该统一为注册表模式，支持按文档类型选择不同的分块策略。

------

### Retrieval 层

**当前问题：** 职责边界模糊，正在吸收本不属于它的功能。

**工业实践方向：**

Retrieval 层的职责应该严格限定为 **"给定 query，返回排序后的候选 chunks"**。它不应该知道 LLM 的存在，不应该做语义理解。

```
Retrieval 层的合法职责：
  ✅ Query 改写 / 扩展（不依赖 LLM 的改写，如分词、同义词）
  ✅ 多路召回（向量、BM25、关键词）
  ✅ RRF / 分数融合
  ✅ Rerank（Cross-Encoder 或基于分数的重排）
  ✅ 元数据过滤
  ✅ 去重

Retrieval 层不应承担的职责：
  ❌ HyDE（需要调用 LLM 生成假设文档）
  ❌ 语义压缩（需要调用 LLM 提取关键句）
  ❌ 相关性验证（需要调用 LLM 判断相关性）
  ❌ Parent Document 替换（这是上下文构建的逻辑）
```

**架构建议：**

- Retrieval 层的对外接口 `IRetrievalService` 应该保持极简：`retrieve(query, top_k, filters) -> list[ScoredChunk]`
- 如果 HyDE 必须用 LLM，那它应该是一个 **Query 预处理步骤**，放在 Orchestration 层调用 Retrieval 之前，或者放在 Generation 层的上下文构建管线中
- Retrieval 内部的多路召回应该用 **Strategy 模式**：`VectorRecallStrategy`、`BM25RecallStrategy`、`KeywordRecallStrategy`，通过配置组合，而非硬编码

------

### Generation 层

**当前问题：** 过于"薄"——只是把 chunks 塞进 prompt 然后调 LLM，缺乏对上下文质量的主动控制。

**工业实践方向：**

Generation 层应该是 RAG 系统中**最"聪明"的层**，它负责：

```
Generation 层的完整职责：
  1. Query 理解与分析（判断 query 类型、复杂度）
  2. 上下文构建管线（上面提到的 Context Augmentation Pipeline）
  3. Prompt 工程（模板选择、动态组装）
  4. LLM 调用（含流式、重试、降级）
  5. 输出后处理（引用提取、安全过滤、格式化）
  6. 拒答决策（上下文不足时主动拒绝）
```

**架构建议：**

- 将 Generation 层内部重构为 

  3 个子阶段

  ：

  - **Pre-Generation**：上下文增强管线（压缩、验证、重排）
  - **Core-Generation**：LLM 调用 + 流式输出
  - **Post-Generation**：输出过滤、引用标注、Guardrails

- 每个子阶段独立可配置，通过特性开关控制

- Prompt 模板应该支持 **条件组合**：根据是否有上下文、是否启用引用标注、是否启用拒答，动态组装 prompt，而非维护 N 个硬编码的模板文件

------

### Orchestration 层

**当前问题：** 职责混合——既是 API 网关，又是业务编排器，又是中间件宿主。

**工业实践方向：**

随着系统复杂度增加，Orchestration 层应该内部分离为 **3 个子层**：

```
Orchestration 层内部分层：

┌─────────────────────────────────────────┐
│  API Gateway (路由、认证、限流、CORS)    │  ← 纯 HTTP 层，不含业务逻辑
├─────────────────────────────────────────┤
│  Application Services (用例编排)         │  ← 业务逻辑编排，调用各层接口
│  - DocumentUploadUseCase                │
│  - QuestionAnswerUseCase                │
│  - ExamGenerationUseCase                │
├─────────────────────────────────────────┤
│  Cross-cutting Middleware               │  ← 横切关注点（日志、追踪、监控）
│  - RequestTracing                       │
│  - MetricsCollector                     │
│  - RateLimiter                          │
└─────────────────────────────────────────┘
```

**架构建议：**

- 当前的 `api/documents.py` 里混合了 HTTP 处理（解析请求、返回响应）和业务编排（调 ingestion、调 retrieval）。应该把业务编排逻辑抽到 `ApplicationService` 中，API 层只做"接收请求 → 调用 Service → 返回响应"。
- 中间件应该按**职责链模式**组织，每个中间件独立可插拔，通过配置决定启用哪些。
- SPA fallback 路由应该和 API 路由在注册时就物理分离，而非靠运行时条件判断。

------

## 三、横切关注点治理

### 可观测性架构

**当前问题：** 日志、追踪、监控散落在各层，没有统一的可观测性策略。

**工业实践标准（三大支柱）：**

```
┌──────────────────────────────────────────────────────────────┐
│                    Observability Layer                        │
│                                                              │
│  ┌────────────┐  ┌────────────┐  ┌────────────────────────┐  │
│  │   Logging  │  │   Tracing  │  │       Metrics          │  │
│  │ (structlog)│  │ (span/ID)  │  │ (Prometheus/Grafana)   │  │
│  └────────────┘  └────────────┘  └────────────────────────┘  │
│                                                              │
│  采集点：                                                    │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ Ingestion:  文档处理耗时、分块数量、索引写入成功率      │ │
│  │ Retrieval:  召回数量、检索延迟、各路召回命中率          │ │
│  │ Generation: LLM 调用延迟、token 用量、拒答率           │ │
│  │ Pipeline:   端到端延迟、各阶段耗时占比                  │ │
│  │ Business:   用户反馈分布、活跃知识库数、文档总量        │ │
│  └─────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

**架构建议：**

- 可观测性应该是**基础设施层**的能力，通过装饰器或中间件注入，而非侵入业务代码
- 每个层的 Facade 方法应该自动被 tracing 装饰器包裹（通过 AOP 或装饰器模式），不需要在每个方法里手动写 span
- Metrics 采集点应该在**设计阶段就确定**，而非事后补。建议为每个层的 Facade 接口定义标准的 metrics 埋点规范

------

### 安全与防护架构

```
┌──────────────────────────────────────────────────────────────┐
│                   Security & Guardrails                       │
│                                                              │
│  输入侧：                                                    │
│  ├── Prompt Injection 检测（规则 + 分类器）                  │
│  ├── 请求体大小限制                                          │
│  ├── Rate Limiting（IP / User / API Key）                    │
│  └── 输入内容合规检查                                        │
│                                                              │
│  输出侧：                                                    │
│  ├── LLM 输出安全过滤                                        │
│  ├── PII（个人身份信息）脱敏                                 │
│  ├── 引用溯源验证（生成的引用是否真实存在于上下文中）        │
│  └── 输出格式合规检查                                        │
│                                                              │
│  数据侧：                                                    │
│  ├── 知识库访问权限隔离（多租户）                            │
│  └── 文档内容敏感信息标记                                    │
└──────────────────────────────────────────────────────────────┘
```

**架构建议：**

- Guardrails 不应该是 Generation 层的内部模块，而应该是**横切 Generation 层的独立切面**——输入检查和输出检查分别在 LLM 调用的前后执行
- 建议设计 `IGuardrail` 接口，支持链式调用多个检查器，每个检查器独立可开关
- 安全防护的降级策略必须明确：检查服务不可用时，是放行还是拒绝？（建议：安全相关的拒绝，非安全相关的放行）

------

## 四、评估闭环架构

**当前问题：** 评估是离线的、孤立的，与生产系统脱节。

**工业实践方向——数据飞轮：**

```
┌─────────────────────────────────────────────────────────────┐
│                    Evaluation Flywheel                        │
│                                                              │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐              │
│  │  线上服务 │───→│ 日志/指标 │───→│ 问题发现 │              │
│  └──────────┘    └──────────┘    └──────────┘              │
│       ↑                                    │                │
│       │                                    ▼                │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐              │
│  │ 灰度发布 │←───│ 策略调优 │←───│ 离线评估 │              │
│  └──────────┘    └──────────┘    └──────────┘              │
│       ↑                                    │                │
│       │                                    ▼                │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐              │
│  │ A/B 测试 │←───│ 对比分析 │←───│ 用户反馈 │              │
│  └──────────┘    └──────────┘    └──────────┘              │
└─────────────────────────────────────────────────────────────┘
```

**架构建议：**

- 评估子系统不应该只是一个"测试工具"，而应该设计为**可持续运行的评估基础设施**
- 线上服务应该能够导出"检索日志"（query + retrieved chunks + scores + final answer + user feedback），供离线评估使用
- Golden Dataset 应该有版本管理，每次评估结果应该持久化，支持历史对比
- A/B 测试框架的核心不是"跑两组配置"，而是**流量分配 + 指标对比 + 统计显著性检验**——这需要一个轻量的实验管理模块

------

## 五、配置与特性管理架构

**当前问题：** 21 个任务将引入大量新配置项，散落在各处，缺乏统一管理。

**工业实践方向：**

```
配置管理的三个层次：

┌─────────────────────────────────────────────────────────────┐
│  Layer 1: 静态配置 (.env / config.yaml)                      │
│  ├── 基础设施配置（DB连接、Redis地址、API密钥）              │
│  ├── 模型配置（Embedding模型、LLM模型、参数）               │
│  └── 全局默认值（chunk_size、top_k、超时时间）              │
├─────────────────────────────────────────────────────────────┤
│  Layer 2: 特性开关 (Feature Flags)                           │
│  ├── 功能开关（enable_hyde、enable_compression）            │
│  ├── 灰度控制（按用户/知识库/百分比）                       │
│  └── 降级开关（emergency_disable_llm_rerank）               │
├─────────────────────────────────────────────────────────────┤
│  Layer 3: 运行时参数 (Per-Request Parameters)               │
│  ├── API 请求中传入的参数（chunk_size、top_k）              │
│  ├── 知识库级别的配置（每个知识库独立的检索策略）            │
│  └── 用户级别的偏好                                        │
└─────────────────────────────────────────────────────────────┘
```

**架构建议：**

- 所有配置项必须有**明确的层次归属**和**优先级规则**（运行时参数 > 特性开关 > 静态配置 > 代码默认值）
- 特性开关应该支持**动态切换**（不重启服务），初期可以用配置文件热加载，后期可以接入远程配置中心
- 每个知识库应该可以独立配置检索策略（有的知识库适合 HyDE，有的不需要），这需要在数据模型中为 KnowledgeBase 表增加 `retrieval_config` JSON 字段

------

## 六、总结：架构演进优先级

```
Phase 1 — 正骨（修复架构违规）
  ├── 补齐 IIngestionService 接口
  ├── 明确 Post-Retrieval 阶段的层归属
  └── 修复层间跨层直接依赖

Phase 2 — 通脉（建立管线模式）
  ├── Context Augmentation Pipeline（可插拔步骤链）
  ├── Retrieval 多路召回 Strategy 模式
  └── Generation 三阶段分离（Pre/Core/Post）

Phase 3 — 强身（横切关注点）
  ├── 统一可观测性层（Metrics + Tracing 装饰器化）
  ├── Guardrails 切面化
  └── 配置管理三层分离

Phase 4 — 闭环（评估与迭代）
  ├── 评估基础设施化（日志导出 + 版本化 Golden Dataset）
  ├── A/B 测试框架
  └── 用户反馈 → 检索策略优化闭环
```

------

> **一句话总结：你当前的四层骨架是对的，但在"长肉"的过程中，最大的架构风险是 Retrieval 层膨胀和层间耦合退化。解决之道是引入"可插拔管线"模式和严格的"层职责边界"——Retrieval 只管找，Generation 管理解和加工，Orchestration 管编排和治理。**
